# WIReader — Как вести проект в Claude Code
> Рабочая методология вайбкодинга | Pro plan (Sonnet) | основано на актуальных доках Claude Code

---

## 0. Базовый факт про Pro-план

- Claude Code на Pro = **только Sonnet** (Opus недоступен, он в Max)
- ~45 промптов в **5-часовом скользящем окне** + недельный потолок
- Лимит **общий** для Claude Code + Claude.ai + Cowork
- Сейчас действует временное повышение недельных лимитов на 50% (до 13 июля 2026)

**Вывод для тебя:** планирование в Claude.ai и кодинг в Claude Code тратят один и тот же бюджет. Но разделять их всё равно нужно — не ради экономии бюджета, а потому что планирование в Claude.ai не тащит весь контекст кодовой базы, а значит каждый промпт там дешевле и чище.

## 1. Операционная модель: два уровня

Ты уже работаешь так на Flarmo, закрепим принцип:

```
┌─────────────────────────────────────────────────────┐
│  УРОВЕНЬ 1 — Claude.ai (веб)         "Мозг"          │
│                                                     │
│  • Планирование архитектуры (готово)                │
│  • Разбор сложных решений ДО кодинга                │
│  • Ревью кода когда что-то идёт не так              │
│  • Опционально: Opus для критичных мест             │
│                                                     │
│  Не держит контекст кодовой базы → дёшево и чисто   │
└─────────────────────────────────────────────────────┘
                        │
                        │ готовая задача / решение
                        ▼
┌─────────────────────────────────────────────────────┐
│  УРОВЕНЬ 2 — Claude Code (терминал)  "Руки"          │
│                                                     │
│  • Исполнение задач из Task Breakdown               │
│  • Одна задача = одна фокусная сессия               │
│  • Sonnet, держит контекст проекта                  │
│                                                     │
│  Дорогой контекст → беречь, не засорять             │
└─────────────────────────────────────────────────────┘
```

Правило: **думать на уровне 1, делать на уровне 2.** Не проси Claude Code проектировать архитектуру на ходу — это сжигает контекст и даёт худший результат.

## 2. CLAUDE.md — операционная система проекта

CLAUDE.md загружается в начало КАЖДОЙ сессии и имеет высокий приоритет — выше чем твои сообщения в чате. Это не документация, это правила поведения.

**Принцип: коротко.** Если не можешь пробежать глазами за минуту — слишком длинно. Двухуровневая структура: контракт (10-20 строк жёстких правил) сверху, детали ниже.

Положи этот файл в корень проекта как `CLAUDE.md`:

```markdown
# WIReader — iOS Reading App

## Что это
iOS-читалка (EPUB/PDF/TXT/FB2) с AI-фичей "Who is?" — справка по
персонажу строго по прочитанному. SwiftUI + MVVM, iOS 17+.

## КОНТРАКТ (не нарушать без явного разрешения)
- SwiftData модели под CloudKit: ВСЕ свойства optional или с default,
  @Attribute(.unique) на id. Иначе краш на синке.
- API-ключи: ТОЛЬКО Keychain. Никогда не хардкод, никогда не UserDefaults.
- Настройки ридера: @AppStorage, НЕ SwiftData.
- RAG-фильтрация позиции: (chapterIndex, chunkIndex), НЕ characterOffset.
- ViewModels: @Observable, НЕ ObservableObject.
- Только async/await. Нет Combine, нет completion handlers.
- Каждая задача должна компилироваться перед переходом к следующей.
- Не добавляй зависимости кроме ZIPFoundation без спроса.

## Стек
- SwiftUI + MVVM + Repository
- SwiftData (2 ModelConfiguration: synced CloudKit + local для AIChunk)
- iCloud Ubiquitous Container — файлы книг
- WKWebView (EPUB) / TextKit 2 (TXT, FB2) / PDFKit (PDF)
- OpenRouter, модель anthropic/claude-sonnet-4-5
- StoreKit 2 — подписки

## Структура
Core/ Data/ Services/ Features/ Resources/
Каждая фича: Views/ + ViewModels/
Репозитории: @MainActor.
Имена файлов: PascalCase = имя типа.

## Сборка
- Открыть в Xcode, target: iOS 17 симулятор
- [сюда добавишь команды если соберёшь CLI-сборку]

## Где что искать
@docs/PRD.md
@docs/Architecture.md
@docs/TaskBreakdown.md
```

**Про `@docs/...`** — синтаксис импорта. Положи наши три документа в папку `docs/` внутри проекта, и Claude Code подтянет их по ссылке когда нужно, не держа в контексте постоянно.

**Auto memory.** Claude Code (v2.1.59+) сам копит заметки между сессиями: команды сборки, инсайты по отладке, твои поправки. Включено по умолчанию. Управление — команда `/memory` в сессии. Не отключай — это бесплатная память о граблях, на которые ты уже наступал.

**Когда дополнять CLAUDE.md:** Claude Code совершил одну и ту же ошибку дважды → запиши правило. Ты второй раз печатаешь одну и ту же поправку → в CLAUDE.md.

## 3. Цикл работы над одной задачей

Это твоя основная петля. Повторяется 40 раз (по числу задач):

```
1. ОТКРЫТЬ задачу в Task Breakdown
   └─ прочитать "Что делаем", "Требования", "Готово когда"

2. /clear  (если предыдущая задача закрыта — чистим контекст!)

3. ДАТЬ задачу Claude Code
   └─ скопировать блок задачи как контекст
   └─ для сложных — сначала "сначала покажи план, не пиши код"

4. PLAN MODE (для нетривиальных задач)
   └─ Claude Code показывает план → ты проверяешь → одобряешь

5. ИСПОЛНЕНИЕ
   └─ Claude Code пишет код

6. ПРОВЕРКА по критерию "Готово когда"
   └─ компилируется? работает как описано?
   └─ НЕТ → поправка в той же сессии
   └─ ДА → дальше

7. COMMIT (git) — фиксируем рабочее состояние

8. Отметить [x] в Task Breakdown
```

**Ключевое: шаг 2 (`/clear`).** Между задачами чисти контекст. Задача 1.4 не должна тащить контекст задачи 1.3. Свежий контекст = выше качество + меньше расход. Это самый недооценённый приём.

## 4. Управление контекстом и лимитами

Главный враг вайбкодинга — раздувание контекста. Длинная сессия → контекст забивается → качество падает → авто-компакция теряет нить.

**Приёмы по убыванию важности:**

1. **`/clear` между задачами.** Закрыл задачу — почистил. Не копи мусор.

2. **Одна задача = одна сессия.** Не делай три задачи подряд в одном контексте. Исключение — мелкие связанные правки.

3. **Plan mode для сложного.** Перед большой задачей: "покажи план, код не пиши". Дёшево поймать кривой подход до того как написана 1000 строк.

4. **Не давай читать лишнее.** Если задача про один файл — не проси Claude Code "осмотреть проект". Точечно.

5. **Следи за процентом контекста.** Claude Code показывает заполнение. Подходит к 70% в середине задачи — заканчивай мысль и `/clear`, не жди авто-компакции.

6. **Бюджет промптов.** ~45 на 5 часов. Одна аккуратная задача с планом и парой правок = 5-10 промптов. То есть 5-8 задач за окно. Phase 1 (10 задач) — два окна.

**Анти-паттерн:** болтать с Claude Code, гонять туда-сюда без плана, держать огромный контекст "на всякий случай". Так лимит улетает за час, а на выходе каша.

## 5. Skills — какие стоит иметь

Skill = переиспользуемая инструкция "как делать X", срабатывает по контексту, стоит ~100 токенов пока не вызвана. Дёшево держать.

Claude Code идёт со встроенными: `/simplify`, `/batch`, `/debug`, `/loop`, `/claude-api`.

**Для твоего проекта стоит создать (project scope, в `.claude/skills/`):**

| Skill | Зачем |
|-------|-------|
| `swift-review` | Проверка кода на соответствие контракту: optional-свойства моделей, async/await вместо Combine, Keychain для ключей |
| `swiftdata-check` | Узкий чеклист по SwiftData+CloudKit граблям перед коммитом модели |
| `commit` | Твой формат коммитов — чтобы история была чистой |

Создаются командой в Claude Code или вручную: папка `.claude/skills/swift-review/SKILL.md` с YAML-заголовком (name, description) и телом-инструкцией. Description — это триггер, пиши как правило маршрутизации ("используй при ревью Swift-кода перед коммитом").

**Не плоди скиллы заранее.** Заведи `swift-review` сейчас, остальные — когда поймёшь что повторяешь одну инструкцию.

## 6. Subagents — какие стоит иметь

Subagent = отдельный экземпляр Claude со своим контекстом, промптом и набором инструментов. Главная ценность — **изоляция контекста**: тяжёлая побочная задача не раздувает основную сессию.

Правило: subagent для **ограниченной** задачи, которая возвращает краткий результат. Не делай "developer-agent который умеет всё" — это просто второй generalist с лишней путаницей.

**Для твоего проекта реально полезны два:**

| Subagent | Когда |
|----------|-------|
| `code-reviewer` | После задачи: свежим взглядом читает диф на баги и нарушения контракта. Свой контекст → не засоряет основной |
| `swiftdata-auditor` | Точечно: проверить модель/миграцию на CloudKit-совместимость |

**Важно про Pro и subagents:** каждый subagent — это отдельный контекст и расход токенов. На Pro-бюджете не злоупотребляй. Используй `code-reviewer` на важных задачах (модели, AI-слой, подписки), а не на каждой мелочи. Для простого багфикса агенты не нужны — это оверинжиниринг.

**Agent teams** (несколько агентов, которые переговариваются) — для тебя избыточно и дорого. Пропусти.

## 7. Главные проблемы вайбкодинга и решения

Это то, ради чего ты спрашивал. Опытный взгляд на грабли:

### Проблема 1: Контекст-дрифт
Сессия длинная, Claude забывает ранние решения, начинает противоречить сам себе.
**Решение:** `/clear` между задачами + CLAUDE.md держит инварианты + auto memory копит инсайты.

### Проблема 2: Claude уходит от плана
Просил одно — сделал пять, переписал то что работало.
**Решение:** атомарные задачи с критерием "Готово когда". Plan mode перед стартом. Если поехал не туда — стоп, `/clear`, переформулировать.

### Проблема 3: Накопление сломанного состояния
Три задачи подряд, в конце ничего не компилируется, непонятно что сломало.
**Решение:** компиляция + git commit после КАЖДОЙ задачи. Сломалось — знаешь виновника, откатываешь одну задачу, а не день работы.

### Проблема 4: Оверинжиниринг
Claude добавляет абстракции, паттерны, "на будущее" — то что не просили.
**Решение:** в CLAUDE.md правило "не добавляй того, чего нет в задаче". `/simplify` если уже наворочено. Ревью с вопросом "это правда нужно сейчас?".

### Проблема 5: Галлюцинации API и устаревшие паттерны
Claude уверенно пишет код с несуществующим или устаревшим API (как было с нашей архитектурой — SwiftData+CloudKit грабли).
**Решение:** для незнакомых/новых фреймворков — проси сослаться на доку, не верь на слово. Критичные места (SwiftData+CloudKit, StoreKit 2, WKWebView bridge) сверяй здесь, в Claude.ai, перед тем как Claude Code их пишет. Опус для таких сверок если есть доступ.

### Проблема 6: Несогласованность между сессиями
Разные сессии делают одно и то же по-разному.
**Решение:** CLAUDE.md фиксирует конвенции. `code-reviewer` ловит отклонения.

### Проблема 7: Слив лимитов
Болтовня, огромный контекст, агенты на каждый чих.
**Решение:** всё из §4. Дисциплина контекста = и качество, и экономия одновременно.

## 8. Git-дисциплина

Не пропускай, даже для пет-проекта. Это твоя страховка.

```
- git init на старте (Phase 0)
- ветка main = всегда рабочее состояние
- коммит после каждой закрытой задачи
- формат: "Phase 1.4: EPUBParser — извлечение метаданных и глав"
- для рискованных задач — отдельная ветка, мердж после проверки
- .gitignore: не коммить API-ключи, .env, DerivedData
```

Claude Code умеет коммитить сам. Но ревью дифа перед коммитом — на тебе. Не коммить вслепую.

## 9. Конкретный старт (твой День 1)

Чтобы не было паралича перед чистым проектом:

```
1. Pre-flight чеклист (bundle ID, название, container ID) — 10 мин
2. Xcode: создать проект (Задача 0.1)
3. git init + .gitignore
4. Положить CLAUDE.md в корень, docs/ с тремя нашими файлами
5. Открыть Claude Code в папке проекта
6. Проверить /memory — убедиться что CLAUDE.md загрузился
7. Дать Задачу 0.2 (capabilities)
8. ... дальше по Task Breakdown
```

Первые задачи (Phase 0) — простые, на них набьёшь руку с циклом §3 без риска.

## Сводка: золотые правила

1. **Думать в Claude.ai, делать в Claude Code**
2. **`/clear` между задачами** — самый важный приём
3. **Одна задача = одна сессия = один коммит**
4. **Компилируется перед переходом дальше**
5. **CLAUDE.md короткий, но с жёстким контрактом**
6. **Критичные технические места сверять до кодинга**
7. **Subagents и skills точечно, не на каждый чих** (Pro-бюджет)
8. **Дисциплина контекста = качество и экономия одновременно**
