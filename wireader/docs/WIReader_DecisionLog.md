- Статусы: `принято` | `заменено` | `под вопросом`.

## A. Scope и продукт

### A1. Форматы: EPUB, PDF, TXT, FB2 — все четыре с MVP
**Почему:** покрыть популярные форматы сразу, иначе теряем аудиторию.
**Статус:** принято

### A2. PDF — только просмотр as-is, без рефлоу текста
**Почему:** в PDF нет понятия текстового потока, рефлоу под экран почти нереален.
**Отвергнуто:** рефлоу PDF (слишком сложно и ненадёжно).
**Статус:** принято

### A3. PDF исключён из AI-обработки
**Почему:** без нормального текстового слоя RAG по PDF ненадёжен. AI только для текстовых форматов.
**Статус:** принято

### A4. Импорт: Files, iCloud Drive, Share Sheet
**Почему:** стандартные нативные пути импорта, покрывают большинство сценариев.
**Отложено:** импорт по URL → v2 (узкий сценарий, неясная польза).
**Статус:** принято

### A5. Режимы чтения MVP: вертикальный скролл + постраничное перелистывание
**Почему:** две базовые механики покрывают основные предпочтения.
**Отложено:** прочие режимы — по мере надобности.
**Статус:** принято

### A6. Автоскролл и автоперелистывание — в бесплатной версии
**Почему:** это базовая функция чтения, а не "удобство сверху". Политика freemium: не запирать базу.
**Статус:** принято

### A7. Прогресс (общий + в главе), закладки, заметки — база
**Почему:** ожидаемый минимум читалки.
**Статус:** принято

### A8. Библиотека: полки, коллекции, теги, поиск
**Почему:** организация растущей библиотеки + поиск обязателен.
**Статус:** принято

### A9. Цели и челленджи — только личные, без соцсоревнования
**Почему:** межпользовательское соперничество порождает накрутки и смещает фокус с чтения на борьбу за место в рейтинге — это убивает миссию "пропаганда чтения".
**Отвергнуто:** лидерборды, друзья, публичные челленджи (требуют бэкенда И вредят идее).
**Статус:** принято

### A10. Статистика: по дням + детализация неделя/месяц/год — обязательно
**Почему:** ключевой инструмент стимуляции регулярного чтения.
**Статус:** принято

### A11. Онбординг обязателен
**Почему:** объяснить киллерфичи и предложить trial при первом запуске.
**Статус:** принято

### A12. TTS (озвучка) — отложено
**Почему:** бесплатные голоса по качеству плохи, качественные стоят дорого; бюджета на это нет.
**Отвергнуто на сейчас:** даже системный AVSpeechSynthesizer отложили до прояснения приоритетов.
**Статус:** отложено

### A13. Перевод/словарь — системные Apple, не AI (для v1)
**Почему:** системных словарей достаточно, не тратим AI-бюджет; пересмотрим с развитием AI-возможностей.
**Статус:** принято (для v1)

## B. Архитектура и платформа

### B1. iOS 17+ минимум (не 16)
**Почему:** открывает SwiftData (нативный CloudKit-синк, чище Core Data). Доля iOS 17+ в 2026 высокая; ради ~3-5% на iOS 16 не тащим сложность на весь проект.
**Отвергнуто:** iOS 16 + Core Data.
**Статус:** принято

### B2. SwiftUI + MVVM + Repository
**Почему:** чистое разделение, тестируемость, консистентно с другими проектами.
**Статус:** принято

### B3. SwiftData (не Core Data)
**Почему:** современный, нативная интеграция с CloudKit, меньше boilerplate.
**Статус:** принято

### B4. Три уровня хранения
- UserDefaults/@AppStorage — настройки ридера
- iCloud Ubiquitous Container — файлы книг
- SwiftData+CloudKit — структурированные данные
**Почему:** каждый тип данных в подходящем хранилище. Файлы синхронятся как файлы, данные — через CloudKit.
**Статус:** принято

### B5. Два ModelConfiguration: synced (CloudKit) + local (AIChunk)
**Почему:** RAG-чанки (тысячи записей на книгу) синхронизировать дорого и бессмысленно — они перестраиваются из файла. Остальное синхронизируется.
**Отвергнуто:** синхронизация чанков через CloudKit.
**Статус:** принято

### B6. SwiftData-модели под CloudKit: все свойства optional/с default, @Attribute(.unique) на id
**Почему:** требование CloudKit — иначе краш на синке. (Поймано на ревью архитектуры.)
**Статус:** принято (ЖЁСТКОЕ ПРАВИЛО)

### B7. Настройки ридера — @AppStorage, не SwiftData
**Почему:** это простые key-value, не реляционные данные. SwiftData избыточен.
**Статус:** принято

### B8. Рендеринг: EPUB→WKWebView, TXT/FB2→TextKit 2, PDF→PDFKit
**Почему:** EPUB это HTML — WKWebView стандарт индустрии. TXT/FB2 нативно через TextKit 2. PDF через Apple-нативный PDFKit.
**Отвергнуто:** нативный рендер EPUB (изобретать велосипед для HTML).
**Статус:** принято

### B9. EPUB загружается через loadFileURL(_:allowingReadAccessTo:)
**Почему:** загрузка HTML-строкой не подтянет относительные ассеты (CSS, картинки). Нужна распакованная директория. (Поймано на ревью.)
**Статус:** принято

### B10. Sign in with Apple — для идентификации в приложении, отдельно от синхронизации
**Почему:** CloudKit работает через iCloud-аккаунт устройства автоматически, SiwA для синка не нужен. SiwA — для раздела "Аккаунт". (Поймано на ревью — концепции были перепутаны.)
**Отложено:** в конец Phase 5 (ничего не блокирует).
**Статус:** принято

### B11. Фоновое индексирование через BackgroundTasks (BGProcessingTask)
**Почему:** RAG-индексирование больших книг долгое — выносим в фон.
**Статус:** принято

### B12. Единственная внешняя зависимость — ZIPFoundation
**Почему:** минимум зависимостей = минимум боли в поддержке. Всё прочее — Apple frameworks.
**Отвергнуто:** Alamofire, Combine, RxSwift, сторонние EPUB-киты.
**Статус:** принято

### B13. @Observable + async/await, без Combine и completion handlers
**Почему:** современный стек iOS 17+, чище.
**Статус:** принято

## C. AI

### C1. AI через OpenRouter (OpenAI-совместимый), не прямой Anthropic API
**Почему:** Anthropic и OpenRouter — разные форматы запросов/заголовков, нельзя просто сменить baseURL. OpenRouter OpenAI-совместим, поддерживает Claude-модели, уже знаком, даёт гибкость смены модели. (Поймано на ревью — в черновике было ошибочное "configurable baseURL".)
**Отвергнуто:** прямой Anthropic API как единственный клиент.
**Статус:** принято

### C2. Модель: anthropic/claude-sonnet-4-5
**Статус:** принято

### C3. AI-клиент за протоколом AIAPIClientProtocol
**Почему:** смена Claude → Apple Foundation Models (если откроют API) = замена одного файла, не рефакторинг.
**Статус:** принято

### C4. RAG-позиция: (chapterIndex, chunkIndex), не characterOffset
**Почему:** EPUB-главы — отдельные HTML-файлы, единого потока символов нет. Абсолютный offset через главы хрупок. (Поймано на ревью.)
**Отвергнуто:** characterOffset как абсолютная позиция.
**Статус:** принято (ЖЁСТКОЕ ПРАВИЛО)

### C5. AI отвечает только по тексту книги до текущей позиции, без общих знаний
**Почему:** общие знания = спойлеры. Юзер должен знать о персонаже только то, что мог прочесть.
**Статус:** принято

### C6. "Who is?" — через контекстное меню (долгий тап), попап-оверлей, НЕ отдельный экран
**Почему:** никаких чатов с AI; переход на отдельный экран сбивает концентрацию и темп чтения.
**Отвергнуто:** чат-интерфейс, отдельный экран.
**Статус:** принято

### C7. Саммари главы — по запросу, кэшируется
**Почему:** не каждый хочет саммари; экономит API-бюджет; исключает авто-спойлер дочитанной главы; юзер контролирует. Генерируется раз, сохраняется (синхронизируется).
**Отвергнуто:** автогенерация при дочитывании главы.
**Статус:** принято

### C8. AI-ответы стримятся по токенам
**Почему:** юзер видит ответ в реальном времени, лучше ощущение.
**Статус:** принято

### C9. AI knowledge base строится при импорте + фоновая доиндексация
**Статус:** принято

### C10. API-ключ — только Keychain
**Почему:** безопасность; никогда не хардкод/UserDefaults.
**Статус:** принято (ЖЁСТКОЕ ПРАВИЛО)

### C11. За API на старте платит разработчик
**Почему:** для тестов/разработки/первых юзеров — OpenRouter или оплаченный Claude API. Пересмотр при росте.
**Статус:** принято

## D. Монетизация

### D1. Freemium: бесплатная база самодостаточна для чтения
**Почему:** миссия — пропаганда чтения. Платное добавляет удобства, не устраняет искусственные ограничения.
**Статус:** принято (ПРОДУКТОВЫЙ ПРИНЦИП)

### D2. Бесплатно: все форматы, все режимы чтения, автоскролл, синк файлов+прогресса, статистика, личные цели
**Почему:** всё необходимое и достаточное для полноценного чтения.
**Статус:** принято

### D3. Платно: AI-фичи, premium-темы/шрифты, расширенная статистика
**Почему:** удобства сверху базы.
**Статус:** принято

### D4. Синхронизация файлов — бесплатно
**Почему:** в 2026 это базовая ожидаемая функция читалки, а не "удобство".
**Статус:** принято

### D5. Два уровня подписки: месяц / год (год со скидкой). Trial 14 дней
**Почему:** 14 дней мало чтобы прочитать всё, достаточно чтобы оценить преимущества.
**Отвергнуто:** несколько уровней подписки (базовая/про) — только free + paid.
**Статус:** принято

## E. Воркфлоу разработки

### E1. Два уровня: Claude.ai (планирование/ревью) + Claude Code (исполнение)
**Почему:** думать там где дёшево и чисто, делать там где есть контекст кода.
**Статус:** принято

### E2. Claude Code на Pro = только Sonnet
**Почему:** Opus доступен только в Max. Факт плана (проверять, меняется).
**Статус:** принято

### E3. Гибрид: нативный агент Xcode (UI-задачи, визуальная верификация превью) + терминальный CLI (всё прочее)
**Почему:** Xcode-агент видит SwiftUI Previews и итерирует визуально — экономит итерации на UI. CLI — последняя версия, кросс-проектная работа, дисциплинированный воркфлоу.
**Статус:** принято

### E4. Цикл: одна задача → /clear → план для сложного → исполнение → проверка → commit
**Почему:** свежий контекст = качество + экономия; компиляция и коммит после каждой задачи = не копим сломанное.
**Статус:** принято

### G1. @Query в View, не массив в ViewModel
**Почему:** идиоматичный SwiftData-паттерн — View реагирует на
изменения базы автоматически без ручного обновления массива.
**Статус:** принято | **Дата:** 9 июня 2026

### G2. UUID вместо Book через Task boundary (Sendable)
**Почему:** Book — reference type без Sendable. UUID — value type,
безопасно передаётся через async boundary. Паттерн для всех
аналогичных случаев в проекте.
**Статус:** принято | **Дата:** 9 июня 2026

### G3. ProgressRepository.updateProgress принимает Book, не bookId
**Почему:** SwiftData-связь book.progress не устанавливается
автоматически по совпадению bookId — нужно явное присвоение
book.progress = record. Без этого book.progress всегда nil.
**Статус:** принято | **Дата:** 9 июня 2026

### G4. Несколько .fileImporter на одном view-дереве — срабатывает последний
**Почему:** SwiftUI-ограничение. Решение: вешать на разные узлы
дерева. Правило добавлено в CLAUDE.md.
**Статус:** принято | **Дата:** 9 июня 2026

### G5. Services/Import/Models/ — поддиректория для ParsedBook, EPUBChapter
**Почему:** временные value types парсинга не относятся к Data/Models/
(SwiftData-модели). Логично держать рядом с парсерами.
Отклонение от Architecture §11 — архитектурный документ не обновляем,
директория де-факто принята.
**Статус:** принято | **Дата:** 9 июня 2026

### G6. TXTParser использует EPUBChapter — архитектурный хак
**Почему:** сделано для компиляции стаба. EPUBChapter.fileURL
подразумевает HTML для WKWebView; TXT рендерится через TextKit 2.
**Отвергнуто:** оставлять как есть в Phase 2.
**Действие:** в задаче 2.1 ввести форматонезависимую модель главы
или убрать зависимость TXTParser от EPUBChapter.
**Статус:** заменено → G8 | **Дата решения:** 9 июня 2026

### G8. Форматонезависимая модель главы: BookChapter + ChapterContent
**Почему:** TXTParser и FB2Parser возвращают плоский текст, EPUB — HTML-файл
для WKWebView. Единая модель EPUBChapter (fileURL) была хаком (G6): TXT не
имеет HTML-файла, концепция fileURL не применима. Вводим `BookChapter`
(index, title?, content) и `ChapterContent` (.html(fileURL:) | .plainText(String)).
Это единственная модель главы во всём проекте; EPUBChapter удалён полностью.
**Отвергнуто:**
- Сохранить EPUBChapter для всех форматов с костылями (рождает WKWebView для TXT).
- Раздельные типы EPUBChapter / TXTChapter / FB2Chapter с общим протоколом
  (протокол избыточен: два варианта enum покрывают всё).
**Технический долг:** RAGIndexer.loadChapters (Phase 3.3) при case .html читает
сырой String(contentsOf:) с HTML-тегами — один огромный чанк. Нужна очистка
HTML перед чанкингом. TODO в коде: TODO(3.3).
**Статус:** принято | **Дата:** 9 июня 2026

### G9. import OSLog обязателен в каждом файле с AppLogger-интерполяцией
**Почему:** Swift-импорты не транзитивны — OSLog не протекает из других файлов.
Без явного import OSLog компилятор не разрешает Logger-интерполяцию. Правило
закреплено в CLAUDE.md.
**Статус:** принято | **Дата:** 9 июня 2026

### G7. RAGRetriever реализован в Phase 1 (опережение)
**Почему:** Claude Code реализовал его превентивно при создании
скелета AI-слоя. Функционально корректен. Задача 3.4 будет лёгкой.
**Статус:** принято | **Дата:** 9 июня 2026
